# 07. 구현 순서

## 한 줄 방향

**먼저 실제로 들어가서 읽고 대화할 수 있게 만들고, 그다음 기록을 책과 장 단위로 차곡차곡 꺼내보게 만든다.**

MVP의 목표는 완성된 서비스처럼 보이는 것이 아니라, 두 사람이 오늘부터 써도 되는 최소한의 흐름을 만드는 것이다.

## 현재 상태

완료된 것:

- 프로젝트 개요 문서
- 디자인 방향 문서
- 기술 스택 문서
- Supabase 프로젝트 연결
- Vercel 프로젝트 연결
- GitHub 저장소 연결
- 목업 파일 `together_app_mockup-1.jsx`
- 성서 seed 데이터 `seed_data.json`

아직 없는 것:

- 실제 Next.js 앱
- Supabase schema migration
- 로그인 구현
- 읽기 상태 저장
- 다음 책 제안/수락
- 코멘트/하이라이트 저장
- 실시간 채팅
- 구절 번호 선택 UI
- 앱 안 알림
- 배포 가능한 첫 화면

## 원칙

- 한 번에 크게 만들지 않는다.
- 배포 가능한 단위로 쪼갠다.
- 매 단계마다 실제로 눌러볼 수 있게 한다.
- 기능보다 데이터 보존을 먼저 안정화한다.
- 자동화보다 수동 운영이 빠르면 처음에는 수동으로 간다.

## Phase 0. 문서와 기준 확정

목표:

- 만들 앱의 범위와 우선순위를 고정한다.

작업:

- `03-data-model.md` 작성
- `04-pages.md` 작성
- `05-features.md` 작성
- `07-roadmap.md` 작성
- 결정 필요 사항 정리
- MVP 범위 확정

완료 기준:

- 7개 루트 문서가 모두 존재한다.
- MVP에 넣을 기능과 미룰 기능이 구분되어 있다.

## Phase 1. Next.js 앱 scaffold

목표:

- Vercel에 올릴 수 있는 실제 앱 뼈대를 만든다.

작업:

- Next.js + TypeScript 프로젝트 생성
- Tailwind CSS 설정
- shadcn/ui 초기 설정
- lucide-react 설치
- Framer Motion 설치
- 기본 라우팅 구성
- 환경 변수 예시 정리
- Vercel Framework Preset을 Next.js에 맞게 조정

추천 구조:

```text
app/
  page.tsx
  login/
  today/
  reading/
  chat/
  records/
components/
lib/
supabase/
```

완료 기준:

- 로컬에서 앱이 실행된다.
- Vercel 배포 화면이 정상적으로 뜬다.
- 랜딩 화면이 목업의 톤을 따른다.

## Phase 2. Supabase schema와 seed

목표:

- 앱의 핵심 데이터를 저장할 DB를 만든다.

작업:

- migration 생성
- `profiles` 테이블 생성
- `sections` 테이블 생성
- `books` 테이블 생성
- `segments` 테이블 생성
- `reading_progress` 생성
- `reading_states` 생성
- `book_proposals` 생성
- `highlights` 생성
- `comments` 생성
- `replies` 생성
- `reactions` 생성
- `messages` 생성
- `message_reads` 생성
- `notifications` 생성
- RLS 활성화
- seed import 스크립트 작성
- `seed_data.json`을 DB에 반영

완료 기준:

- Supabase Dashboard에서 8개 section, 66권, 1,189개 segment가 보인다.
- 주환/희진 profile이 존재한다.
- 앱에서 segment 목록을 읽을 수 있다.

## Phase 3. 로그인

목표:

- 두 사람만 앱에 들어올 수 있게 한다.

작업:

- 랜딩 화면 구현
- 사람 선택 UI 구현
- 비밀번호 입력 화면 구현
- Supabase Auth 연결
- 로그인 세션 유지
- 로그아웃 구현
- 보호된 화면 처리

완료 기준:

- 주환/희진으로 각각 로그인할 수 있다.
- 로그아웃 후 다시 랜딩으로 돌아온다.
- 로그인하지 않으면 내부 화면에 접근할 수 없다.

## Phase 4. 오늘 화면과 읽음 체크

목표:

- 앱을 열면 현재 읽을 장이 보이고, 읽음 체크가 저장된다.

작업:

- 현재 reading_progress 조회
- 현재 segment 표시
- WOL 신세계역 성경 연구용 링크 생성
- 두 사람의 reading_state 조회
- `다 읽었어요` 버튼 구현
- 한쪽만 읽은 상태 표시
- 둘 다 읽었을 때 다음 segment 활성화
- 책 마지막 장 처리

완료 기준:

- 실제 DB에 읽음 체크가 저장된다.
- 두 사람 모두 체크하면 다음 장으로 넘어간다.
- 화면 문구가 부담을 주지 않는다.

## Phase 5. 실시간 채팅

목표:

- 둘만의 연락 수단으로 쓸 수 있는 채팅을 빠르게 만든다.

작업:

- 메시지 목록 조회
- 메시지 전송
- Supabase Realtime subscription
- 날짜 구분선
- 읽음 표시
- message_reads 처리
- Presence channel 연결
- 접속 중 표시
- 메시지 수정/삭제
- 모바일 전체 화면 채팅
- 웹 오른쪽 drawer 또는 floating panel

완료 기준:

- 한쪽에서 보낸 메시지가 상대 화면에 실시간으로 뜬다.
- 읽음 표시가 작게 반영된다.
- 접속 중 표시가 채팅에서만 보인다.
- 웹에서는 버튼으로 채팅을 열고 닫을 수 있다.

## Phase 6. Segment 상세 기록

목표:

- 장별 코멘트와 하이라이트를 실제로 남길 수 있게 한다.

작업:

- segment 상세 화면 구현
- 하이라이트 작성
- 구절 번호 선택 UI
- 코멘트 작성
- 답글 작성
- 반응 토글
- 하이라이트 수정/삭제
- 코멘트 수정/삭제
- 답글 수정/삭제
- 목록 조회
- 과거 segment 접근

완료 기준:

- 특정 segment에 하이라이트와 코멘트를 남길 수 있다.
- 새로고침해도 기록이 남아 있다.
- 두 사용자 색이 구분된다.

## Phase 7. 읽기 화면과 책 제안

목표:

- 현재 책과 과거 segment를 탐색하고, 책을 끝낸 뒤 다음 책을 제안/수락할 수 있게 한다.

작업:

- section별 책 목록
- 책별 segment 목록
- segment 상태 표시
- 현재 segment 강조
- 완료된 segment 재방문
- 시작 책을 전도서로 설정
- 책 완료 상태 처리
- 다음 책 제안
- 상대의 수락
- 수락 후 새 책 1장으로 이동

완료 기준:

- 전도서뿐 아니라 모든 책/장을 탐색할 수 있다.
- 완료된 장의 기록을 다시 볼 수 있다.
- 다음 책을 제안하고 수락할 수 있다.

## Phase 8. 기록 화면

목표:

- 책과 장 단위로 쌓인 기록을 다시 꺼내볼 수 있게 한다.

작업:

- 성경 책 목록
- 책별 읽음 요약
- 책별 장 목록
- 장 옆 읽음 상태 표시
- 장별 읽은 날짜 표시
- 장별 코멘트 표시
- 장별 하이라이트 표시
- 답글과 반응 표시
- 기록에서 segment 상세로 이동
- 앱 안 알림 badge 연결

완료 기준:

- 책을 고르고 장을 고르면 그 장의 기록을 볼 수 있다.
- 각 장이 읽혔는지 바로 알 수 있다.
- 그 장을 언제 읽었는지 확인할 수 있다.
- 새 메시지, 새 답글, 책 제안 같은 변화가 앱 안에서 보인다.

## Phase 9. polish와 안정화

목표:

- 실제로 오래 쓰기 편한 상태로 다듬는다.

작업:

- 모바일 레이아웃 점검
- 웹 레이아웃 점검
- 빈 상태 문구 정리
- 로딩/오류 상태 정리
- 접근성 점검
- 텍스트 대비 점검
- RLS 정책 점검
- 배포 환경 변수 점검
- 기본 테스트 추가

완료 기준:

- 모바일과 웹에서 주요 흐름이 깨지지 않는다.
- 새로고침, 재로그인, 느린 네트워크에서도 기본 흐름이 버틴다.
- Vercel 배포본으로 실제 사용이 가능하다.

## 개발 순서 추천

1. Next.js 앱 생성
2. 디자인 토큰과 기본 레이아웃
3. Supabase schema
4. seed import
5. 로그인
6. 오늘 화면
7. 읽음 체크
8. 채팅
9. segment 상세
10. 하이라이트/코멘트
11. 책 제안/수락
12. 기록
13. 반응형 polish

이 순서가 좋은 이유는, 먼저 "앱에 들어와서 현재 장을 보는 경험"을 만들고 곧바로 연락 수단인 채팅을 붙인 뒤, 기록과 책 선택 흐름을 얹기 때문이다.

## 리스크

### 로그인 UI와 Auth 연결

Supabase Auth를 쓰되 화면은 이름 선택 + 비밀번호 입력으로 보이게 해야 한다.

추천:

- 두 사용자 계정을 미리 만들고, 선택한 이름에 대응되는 내부 이메일로 Supabase Auth 로그인을 수행한다.

### WOL 연구용 성경 링크 정확도

책 번호와 chapter를 WOL 신세계역 성경 연구용 URL에 맞게 생성해야 한다.

추천:

- seed 데이터에 `wol_book_number` 또는 `jw_url`을 명시적으로 넣는다.
- URL 패턴은 `https://wol.jw.org/ko/wol/binav/r8/lp-ko/nwtsty/{book_number}/{chapter}`를 기준으로 한다.
- 전도서 1장 예시는 `https://wol.jw.org/ko/wol/binav/r8/lp-ko/nwtsty/21/1`이다.
- 처음에는 주요 책 몇 개로 검증한 뒤 전체 생성한다.

### Realtime/RLS 조합

Supabase Realtime과 RLS 정책이 맞지 않으면 채팅이 안 보일 수 있다.

추천:

- schema 작성 직후 작은 채팅 prototype으로 먼저 검증한다.

### 기록 화면 과욕

기록 화면은 MVP에 포함하지만, 검색과 고급 필터까지 한 번에 넣으면 늦어진다.

추천:

- 성경 책 목록, 장 목록, 장별 기록 상세까지만 먼저 만든다.

## 결정된 것

- 첫 책은 전도서로 시작한다.
- 기록 탭은 첫 배포에 포함한다.
- 채팅은 Phase 5로 당긴다.
- 다음 책은 제안/수락 흐름으로 정한다.
- 웹 채팅은 버튼으로 여닫는다.
- 구절 번호 선택 UI는 첫 배포에 포함한다.
- 알림은 앱 안에서 확인할 수 있게 만든다.
