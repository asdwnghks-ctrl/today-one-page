# 07. 구현 순서

## 한 줄 방향

**먼저 실제로 들어가서 읽고 대화할 수 있게 만들고, 그다음 기록을 책과 장 단위로 차곡차곡 꺼내보게 만든다.**

MVP의 목표는 완성된 서비스처럼 보이는 것이 아니라, 두 사람이 오늘부터 써도 되는 최소한의 흐름을 만드는 것이다.

## 현재 상태

### MVP 단계에서 완료된 것 (2인 전용 시절)

- Next.js 앱 scaffold와 실제 화면 구현
- Tailwind 기반 반응형 UI
- Supabase schema migration과 seed 데이터 반영
- 주환/희진 이름 선택 로그인
- 오늘 화면과 WOL 본문 링크
- 읽음 체크와 한국시간 오전 2시 기준 다음 장 전환
- 책 회차별 `session_id`, 못 읽음 기록, 선물 약속/공개
- 다음 책 제안/수락, 수락된 책의 예약 전환
- 코멘트, 하이라이트, 답글, 하트 반응
- 구절 번호 선택 UI (전도서만)
- 실시간 채팅, 읽음 표시, presence
- 앱 안 알림과 unread badge
- 기록 탭
- Vercel 배포 연결과 GitHub post-commit push 흐름
- Supabase 원격 migration 적용
- 읽음 체크, 반응, 제안/수락, 선물 저장, 코멘트/하이라이트 작성은 화면 반응을 먼저 보여주고 서버 호출을 뒤따라가게 했다.

### 멀티 그룹 확장 (완료, `08-multi-group.md` 참고)

- 그룹/초대코드/개인 PIN(scrypt) 기반 인증으로 전환. 기존 주환/희진 데이터는 첫 번째 그룹(`TGBHER`)으로 마이그레이션.
- 두 가지 읽기 방식: 하루 1장(`daily_one`) / 읽기 계획표(`plan`, JW 1년 통독표 363일).
- 그룹 생성 시 시작 책(또는 계획표 시작 범위)을 바로 고르도록 해서 빈 상태 없이 시작.
- **채팅 기능 완전 제거** (메시지, 읽음 표시, presence). 기존 대화는 `docs/chat-archive.md`로 보존.
- 다음 책 제안/수락(2인 전용) → 그룹장 지정 + 자동 선택(안 읽은 책 우선, 다 읽으면 재독)으로 전환.
- N명 대응: 못 읽음/선물 로직이 복수 패자를 지원.
- 그룹 정보 화면: 초대코드 재확인(그룹원 전체), 멤버 이름 변경(본인), 그룹 이름 변경(그룹장).
- 절 수(`verse_counts`) 데이터를 성경 전체 1,189개 장으로 확장(기존엔 전도서 12장뿐).
- 선물 공개를 가장 최근 회차로만 제한(예전엔 과거 회차 것도 계속 같이 떴음).
- 기록 탭 기본 선택 책을 "오늘 읽는 책"으로 동기화(예전엔 전도서 고정).
- **1000행 PostgREST 캡 발견 및 수정**: `segments`, `verse_counts`가 1000행을 넘으면서 실제로 데이터가 잘리는 문제를 겪었고, 페이지네이션 헬퍼(`fetchAllRows`)로 해결. 앞으로 큰 테이블을 다룰 때 항상 염두에 둘 것.

남은 개선 후보:

- RLS 정책 세부 점검 (지금은 서버 라우트의 `group_id` 필터링에만 의존)
- 자동 테스트 보강
- 기록 탭 고급 필터
- 그룹장 위임 / 멤버 추방 / 그룹 삭제 UI
- 계획표 여러 종류 지원 (지금은 JW 1년 통독표 1개)
- 웹 푸시/이메일 알림 여부 재검토
- 사용량이 늘 때 pagination과 에러 추적 보강

## 원칙

- 한 번에 크게 만들지 않는다.
- 배포 가능한 단위로 쪼갠다.
- 매 단계마다 실제로 눌러볼 수 있게 한다.
- 기능보다 데이터 보존을 먼저 안정화한다.
- 자동화보다 수동 운영이 빠르면 처음에는 수동으로 간다.

## 리스크

### 로그인 UI와 세션 처리

화면은 이름 선택 + 비밀번호 입력으로 보이게 하고, 서버에서는 공유 PIN과 `top_profile` cookie로 세션을 유지한다.

추천:

- `APP_SHARED_PIN`은 Vercel 환경 변수에만 둔다.
- `top_profile` cookie는 httpOnly, sameSite lax, production secure로 설정한다.
- 나중에 보안 요구가 커지면 Supabase Auth로 옮기는 방안을 다시 검토한다.

### WOL 연구용 성경 링크 정확도

책 번호와 chapter를 WOL 신세계역 성경 연구용 URL에 맞게 생성해야 한다.

추천:

- seed 데이터에 `wol_book_number` 또는 `jw_url`을 명시적으로 넣는다.
- URL 패턴은 `https://wol.jw.org/ko/wol/b/r8/lp-ko/nwtsty/{book_number}/{chapter}`를 기준으로 한다.
- 전도서 1장 예시는 `https://wol.jw.org/ko/wol/b/r8/lp-ko/nwtsty/21/1`이다.
- 처음에는 주요 책 몇 개로 검증한 뒤 전체 생성한다.

### Realtime/RLS 조합

Supabase Realtime과 RLS 정책이 맞지 않으면 채팅이 안 보일 수 있다.

추천:

- schema 작성 직후 작은 채팅 prototype으로 먼저 검증한다.

### 기록 화면 과욕

기록 화면은 MVP에 포함하지만, 검색과 고급 필터까지 한 번에 넣으면 늦어진다.

추천:

- 성경 책 목록, 장 목록, 장별 기록 상세까지만 먼저 만든다.

## 결정된 것

- 첫 책은 전도서로 시작한다.
- 기록 탭은 첫 배포에 포함한다.
- 채팅은 Phase 5로 당긴다.
- 다음 책은 제안/수락 흐름으로 정한다.
- 웹 채팅은 버튼으로 여닫는다.
- 구절 번호 선택 UI는 첫 배포에 포함한다.
- 알림은 앱 안에서 확인할 수 있게 만든다.

- 앱 밖 알림은 실제 사용 후 필요할 때 웹 푸시 또는 이메일 중 하나를 고른다.

## 이후 업데이트 (멀티 그룹 확장, `08-multi-group.md` 참고)

위 "결정된 것"은 MVP(2인 전용) 시점 기준이며, 멀티 그룹 확장 이후 다음이 바뀌었다:

- 로그인: 공유 PIN → 초대코드 + 그룹별 개인 PIN.
- 다음 책: 제안/수락 흐름 → 그룹장 지정 + 자동 선택.
- 채팅: 전부 제거. 웹 채팅 drawer/floating panel 관련 결정은 더 이상 적용되지 않는다.
- 시작 책: 전도서 고정 → 그룹 생성 시 직접 선택.
- 구절 번호 선택 UI: 전도서만 → 전체 66권.
