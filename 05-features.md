# 05. 기능 명세

## 한 줄 방향

**읽기와 기록. 두 기능을 여러 그룹이 각자의 속도로 오래 쌓이게 만든다.**

원래 MVP는 "읽기 + 채팅"이었지만, 채팅은 완전히 제거하고 멀티 그룹 확장에 집중했다. 예쁜 부가 기능보다 로그인, 현재 장, 읽음 체크, 코멘트, 하이라이트가 먼저다.

## 기능 우선순위

### 구현 완료

- 그룹 생성 / 초대코드 참여 로그인
- 개인 PIN (scrypt 해싱)
- 두 가지 읽기 방식: 하루 1장 / 읽기 계획표(1년 통독, 363일)
- 그룹 생성 시 시작 지점 직접 선택 (시작 책 또는 계획표 시작 범위)
- 현재 읽는 segment(또는 계획표 오늘 범위) 표시
- WOL 신세계역 성경 연구용 본문 링크
- 읽음 체크 (계획표 모드는 하루 범위 전체를 한 번에 체크)
- 그룹원 전원의 읽음 상태 (N명 대응)
- segment별 하이라이트 — **전체 책/장 절 번호 선택 UI**
- segment별 코멘트
- 답글
- 작은 반응(하트)
- 다음 책 자동 선택 + 그룹장 수동 지정
- 회차별 못 읽음 기록 (N명 대응, 복수 패자 가능)
- 회차별 선물 약속과 공개 (최근 회차 것만 노출)
- 기록 탭 (오늘 읽는 책 기본 선택)
- 코멘트/하이라이트 수정과 삭제
- 그룹 정보: 초대코드 확인, 멤버 이름 변경(본인), 그룹 이름 변경(그룹장)
- 앱 안 알림 표시
- 그룹별 데이터 완전 격리
- 기본 반응형 레이아웃

### 제거된 것

- **실시간 채팅** (메시지, 읽음 표시, 접속 상태 presence 전부 제거). 기존 채팅 내역은 `docs/chat-archive.md`로 보존.
- **다음 책 제안/수락 협상** (2인 전용 흐름이라 N명 그룹에 안 맞음) → 그룹장 지정 + 자동 선택으로 대체.

### 이후 확장 후보

- 날짜별 회고
- "작년 오늘 우리는"
- 그룹장 위임 / 멤버 추방 / 그룹 삭제 UI
- 계획표 여러 종류 지원 (지금은 JW 1년 통독표 1개만)
- RLS 정책 강화
- Vercel Analytics / 오류 추적(Sentry)

## 1. 그룹 로그인

### 목적

그룹 단위로 격리된 접근을 만든다. 회원가입은 없다.

### 요구사항

- 랜딩에서 "그룹 만들기" 또는 "초대코드로 들어가기"를 고른다.
- 그룹 생성: 그룹명, 내 이름/색상, 읽기 방식, 시작 지점, PIN을 정하면 초대코드가 발급된다.
- 초대코드 참여: 코드를 입력하면 그룹명/멤버 목록이 보이고, 본인 이름이 있으면 PIN 입력, 없으면 새 멤버로 참여한다.
- 로그인 후에는 `top_group`(그룹 id) + `top_profile`(slug) 두 개의 httpOnly cookie로 세션을 유지한다.
- 로그아웃할 수 있어야 한다.

### 정책

- 그룹 인원은 2~5명.
- 같은 그룹 안에서 색상은 중복될 수 없다.
- 비밀번호 찾기 흐름은 없다 — 그룹 정보 화면에서 로그인된 멤버가 초대코드를 다시 확인할 수 있는 게 유일한 복구 수단이다. 전원 로그아웃 + 코드/PIN 분실 시 복구 불가.

### 구현 방식

`lib/pin.ts`가 scrypt로 PIN을 해싱/검증한다. `app/api/groups/route.ts`(생성), `app/api/join/route.ts`(참여), `app/api/groups/lookup/route.ts`(초대코드로 멤버 목록 조회, 인증 불필요), `app/api/login/route.ts`(로그인)가 흐름을 나눠 처리한다.

## 2. 현재 읽기 상태

### 목적

앱을 열었을 때 "지금 그룹이 어디를 읽고 있는지" 바로 알게 한다.

### 요구사항

- 현재 책, 장(계획표 모드는 오늘의 범위 전체)을 보여준다.
- `본문 보기` 버튼은 WOL 신세계역 성경 연구용 해당 장을 새 창으로 연결한다.
- 앱 안에 성서 본문을 저장하거나 노출하지 않는다.
- 그룹원별 읽음 상태를 보여준다.
- 전원이 읽었고 다음 한국시간 오전 2시 경계가 지났으면 다음으로 넘어간다.

### 상태 문구

- 아무도 읽지 않음: "함께 읽는 중"
- 전원 읽음: "오늘은 모두 읽었어요. 새벽 2시에 다음 범위가 열려요."
- 그 외 평범한 상태에는 **문구를 아예 띄우지 않는다** (오글거리는 채움 문구 제거).

### 제외

- 강제 날짜 계획표(하루 1장 모드는)
- 연속 일수
- 전체 퍼센트
- 지각/실패 표현

## 3. 읽음 체크

### 목적

그룹원 전원이 읽어야 다음으로 넘어가는 구조를 만든다.

### 요구사항

- 사용자는 현재 segment(들)에서 `다 읽었어요`를 누를 수 있다.
- 버튼을 누르면 화면에서는 즉시 읽음 상태로 보인다(optimistic update).
- 체크하면 `reading_states.checked_at`이 저장된다. `check_read` 액션은 `segmentIds`(복수, 계획표 모드용)와 `segmentId`(단수, 하위호환) 둘 다 받는다.
- 그룹원 중 아직 안 읽은 사람이 있으면 다음으로 넘기지 않는다.
- 전원이 체크한 뒤 다음 한국시간 오전 2시 경계가 지나면 다음으로 활성화한다.
- 못 읽은 사람이 있고 본인은 다 읽었으면 "다음 범위 열기"(수동 진행)를 쓸 수 있다.

## 4. 다음 책 자동 선택 / 그룹장 지정 (하루 1장 모드)

### 목적

N명 그룹에 안 맞는 "제안/수락" 2인 협상 구조를 없애고, 자동화 + 최소한의 그룹장 권한으로 대체한다.

### 요구사항

- 책이 완료되면 자동으로 다음 책이 정해진다: 그룹의 읽은 기록에 없는 책 중 정렬 순서가 가장 앞선 것. 66권을 다 읽었으면 창세기부터 다시(재독 허용).
- 그룹장은 "다음 책" 카드에서 언제든 원하는 책으로 바꿀 수 있다(`set_next_book` 액션, 그룹장 본인 확인 후 처리).
- 그룹장이 아닌 멤버는 다음 책을 읽기 전용으로만 본다.
- 그룹이 막 만들어졌을 때는 생성 폼에서 시작할 책을 바로 고르므로, 별도의 "다음 책 고르기" 단계 자체가 없다.
- 레거시로 `choosing_book` 상태에 멈춰 있던 그룹도 그룹장이 다음 책을 지정하면 그 자리에서 즉시 시작된다(자가 치유).

## 5. 읽기 계획표 모드

### 목적

정해진 통독 계획표를 따라가고 싶은 그룹을 위한 대안 모드.

### 요구사항

- 그룹 생성 시 `daily_one` 대신 `plan`을 고르면, 11개 범주(모세의 기록~다른 사도와 제자들의 글) 중 시작 지점을 고른다.
- 하루 범위는 계획표에 따라 여러 장(가끔 두 책에 걸쳐)일 수 있다.
- 읽음 체크, 다음날 넘어가는 게이트(전원 체크 + 오전 2시 경계), 못 읽음 기록, 수동 진행 로직은 하루 1장 모드와 동일한 코드 경로를 재사용한다.
- 30 plan-day마다 선물 회차가 끝나고 새 회차가 시작된다(하루 1장의 "책 완료"에 대응하는 개념).
- 363일차를 완료하면 `completed` 상태로 멈춘다(재시작 UI는 없음).
- 이 모드에는 "다음 책" 카드/그룹장 지정 기능이 없다 — 계획표가 순서를 전부 정하기 때문.

## 4-1. 못 읽음 기록과 선물

### 목적

읽지 못한 날을 벌점처럼 보이게 하지 않으면서, 회차를 마쳤을 때 그룹만의 작은 약속을 공개한다.

### 요구사항

- 한국시간 오전 2시를 하루 경계로 사용한다.
- 경계가 지났을 때 현재 범위를 다 읽지 않은 사람을 `reading_misses`에 기록한다.
- 같은 사람/segment/경계 조합은 중복 기록하지 않는다.
- 못 읽음 횟수는 현재 회차의 `session_id` 기준으로 계산한다. **N명 그룹에서 최다 못읽음 인원이 여러 명이면 전부 패자로 처리된다.**
- 각자는 현재 회차에 선물 약속을 하나 적어둘 수 있다.
- 다른 그룹원에게는 선물 내용이 아니라 적어두었는지만 보인다.
- 회차가 끝나면(하루 1장: 책 완료, 계획표: 30일마다) 못 읽음 횟수가 가장 많은 사람(들)의 선물이 공개된다.
- 동률이거나 전원 0회면 전원 공개될 수 있다.
- **화면에는 가장 최근 회차의 선물만 보인다.** 예전 회차의 공개된 선물은 다시 뜨지 않는다(실사용 버그를 고쳤다 — 예전에는 최근 6개를 전체 기간에서 가져와서 옛날 것도 계속 같이 떴다).

### 톤

- 못 읽음 횟수는 경쟁이나 꾸중처럼 보이지 않게 작게 보여준다.
- "실패", "벌칙", "밀림" 같은 표현은 쓰지 않는다.

## 5. 하이라이트

### 목적

마음에 남는 구절을 segment에 묶어 남긴다.

### 요구사항

- 구절 참조를 입력한다.
- **장 안의 구절 번호를 버튼으로 선택할 수 있다 — 이제 성경 전체 66권 1,189개 장에서 동일하게 동작한다.** (예전에는 전도서만 절 수 데이터가 있어서 버튼이 뜨고, 나머지는 직접 타이핑해야 했다. `scripts/seed-verse-counts.ts`로 전체 절 수를 채워서 통일했다.)
- 단일 구절과 구절 범위를 모두 지원한다.
- 색을 선택한다.
- 짧은 메모를 선택적으로 남긴다.
- 작성자 색으로 누가 남겼는지 보여준다.
- 답글과 반응을 달 수 있다.
- 과거 segment에도 하이라이트를 추가할 수 있다.
- 본인 하이라이트는 수정/삭제할 수 있다.

### 색상

하이라이트 색 선택지(파스텔): 핑크, 라벤더, 하늘, 올리브. (멤버 식별 색상 팔레트와는 별개.)

### 절 수 데이터 정확도

표준(대부분의 영어 성경이 따르는) 장별 절 수 기준. NWT와 극소수 장에서 ±1절 차이 가능성이 있으나, jw.org 스터디 바이블 표본 대조로 확인했다. 데이터가 없는 예외 상황에서는 직접 입력 fallback이 여전히 남아있다.

## 6. 코멘트

### 목적

특정 segment에 묶이는 깊은 생각을 저장한다.

### 요구사항

- segment 상세에서 코멘트를 작성한다.
- 작성자, 작성 시간, 본문을 보여준다.
- 답글을 달 수 있다.
- 작은 반응을 달 수 있다.
- 과거 segment에도 새 코멘트를 추가할 수 있다.
- 본인 코멘트는 수정/삭제할 수 있다 (soft delete).

### 정책

- 코멘트에는 읽음 표시나 접속 상태를 붙이지 않는다.

## 7. 답글과 반응

### 목적

긴 코멘트를 새로 쓰지 않아도 작게 반응할 수 있게 한다.

### 요구사항

- 하이라이트/코멘트에 답글을 달 수 있다.
- 하이라이트, 코멘트, 답글에 반응할 수 있다.
- 같은 사람이 같은 대상에 같은 반응을 중복으로 달 수 없다.
- 본인 답글은 수정/삭제할 수 있다.
- 하트 반응은 화면에서 먼저 바뀌고 서버 저장은 뒤따라간다.

## 8. 그룹 정보 관리

### 목적

초대코드를 잃어버려도 로그인만 되어 있으면 다시 확인할 수 있게 하고, 이름을 스스로 관리하게 한다.

### 요구사항

- 헤더의 그룹 정보 아이콘으로 drawer를 연다.
- 초대코드를 큰 글씨로 보여주고 복사 버튼을 제공한다. 그룹원 누구나 볼 수 있다(그룹장 전용 아님).
- 멤버 목록(이름 + 색상)을 보여준다.
- 본인 이름은 누구나 바꿀 수 있다(`update_my_name`).
- 그룹 이름은 그룹장만 바꿀 수 있다(`update_group_name`).

### 정책

- 초대코드 분실 대비책은 "로그인된 멤버가 있으면 이 화면에서 재확인" 뿐이다. 이메일/전화 기반 계정 복구는 없다.

## 9. 기록

### 목적

시간이 지나 쌓인 기록을 다시 보는 공간을 만든다.

### 요구사항

- 성경 책 목록을 보여준다. **기본 선택은 오늘 읽는 책이다** (예전에는 전도서로 고정되어 있었다).
- 책을 누르면 장 목록을 보여준다.
- 각 장 옆에 읽음 상태를 표시한다.
- 장을 누르면 장별 기록 상세로 이동한다.
- 장별 기록 상세에는 읽은 날짜, 코멘트, 하이라이트, 답글, 반응을 보여준다.
- 기록 탭은 열람 전용이다. 새 코멘트 작성, 수정, 삭제는 `코멘트` 탭에서 한다.

## 10. 알림

MVP에서는 앱 밖 푸시 알림은 넣지 않고, 앱 안에서 확인할 수 있는 조용한 알림만 넣는다.

요구사항:

- 새 코멘트/답글 badge
- 다음 책 지정 알림 (그룹장이 지정했을 때, 시작 세그먼트가 열렸을 때)
- 그룹원이 읽음 체크했을 때 앱 안 알림
- 알림 목록에서 관련 화면으로 이동
- 알림 목록에는 읽지 않은 알림만 보여준다.
- (제거됨) 채팅 관련 알림은 더 이상 만들어지지 않고, 상태 조회에서 과거 `message` 타입 알림도 걸러낸다.

## 11. 관리자/운영

공개 관리자 화면은 만들지 않는다.

운영은 Supabase Dashboard + Supabase Management API(SQL 실행)로 한다.

- 그룹/사용자 계정 직접 조회/정리 (막힌 테스트 그룹 삭제 등)
- 현재 책/segment 수정
- 잘못된 데이터 수정
- seed 데이터 확인

## 비기능 요구사항

### 보존

- 코멘트, 하이라이트는 영구 보관한다.
- 삭제는 가능하면 soft delete로 처리한다.

### 보안

- 그룹 간 데이터는 서버 라우트의 `group_id` 필터링으로 격리한다(RLS는 아직 켜지 않음, `03-data-model.md` 참고).
- PIN은 scrypt로 해싱해서 저장한다(평문 저장 없음).
- service role key는 브라우저에 노출하지 않는다.

### 성능

- 초기 화면은 빠르게 떠야 한다.
- 1000행을 넘는 테이블(`segments`, `verse_counts`)은 페이지네이션으로 가져온다(Supabase 호스팅 프로젝트의 PostgREST 1000행 캡 때문).
- 읽음 체크, 반응, 다음 책 지정, 코멘트/하이라이트 작성, 선물 저장은 화면 반응을 먼저 보여주고 서버 저장을 뒤따라가게 한다.
- 서버 호출 중이라는 이유로 전체 화면을 막거나 마우스 커서를 로딩 상태로 바꾸지 않는다.
- 로그인만 예외적으로 버튼 안에 `확인 중` 상태를 표시한다.

### 접근성

- 버튼은 모바일에서 누르기 충분히 커야 한다.
- 작성자 색만으로 의미를 전달하지 않는다.
- 텍스트 대비는 충분히 확보한다.

## 결정된 것

- 코멘트/하이라이트 수정/삭제는 포함한다.
- 기록 탭은 오늘 읽는 책을 기본으로 연다.
- 구절 번호 선택 UI는 전체 책에 적용한다.
- 알림은 앱 안 unread badge와 알림 목록으로 넣는다.
- 채팅은 완전히 제거한다.
- 다음 책은 자동 선택 + 그룹장 지정으로 처리한다(제안/수락 없음).
