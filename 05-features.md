# 05. 기능 명세

## 한 줄 방향

**읽기, 기록, 대화. 세 기능을 작게 시작해서 오래 쌓이게 만든다.**

MVP는 "둘이 실제로 매일 또는 가끔 쓸 수 있는가"를 기준으로 잡는다. 예쁜 부가 기능보다 로그인, 현재 장, 읽음 체크, 코멘트, 하이라이트, 채팅이 먼저다.

## 기능 우선순위

### MVP 필수

- 사람 선택 로그인
- 현재 읽는 segment 표시
- WOL 신세계역 성경 연구용 본문 링크
- 읽음 체크
- 둘의 읽음 상태
- segment별 하이라이트
- segment별 코멘트
- 답글
- 작은 반응
- 실시간 채팅
- 채팅 읽음 표시
- 채팅 접속 상태
- 책 제안/수락
- 기록 탭
- 코멘트/채팅 수정과 삭제
- 구절 번호 선택 UI
- 앱 안 알림 표시
- 기본 반응형 레이아웃

### MVP에 넣으면 좋은 것

- 과거 segment 다시 보기
- 메시지 날짜 구분선
- 부드러운 화면 전환

### MVP 이후

- 날짜별 회고
- "작년 오늘 우리는"
- 채팅 검색
- 웹 푸시 또는 이메일 알림
- 이미지 첨부
- 더 다양한 반응
- Vercel Analytics
- Sentry

## 1. 사람 선택 로그인

### 목적

두 사람만 들어올 수 있게 한다. 회원가입은 만들지 않는다.

### 요구사항

- 랜딩에 주환, 희진 카드가 보인다.
- 카드를 누르면 해당 사람의 로그인 화면으로 이동한다.
- 비밀번호를 입력하면 세션이 생성된다.
- 로그인 후에는 마지막 사용자를 기억해도 된다.
- 로그아웃할 수 있어야 한다.

### 정책

- 외부 가입 링크 없음
- 비밀번호 찾기 흐름은 MVP 제외
- 계정 추가 기능 없음

### 구현 방식

Supabase Auth를 내부적으로 사용한다. 사용자는 이름 카드 선택 후 비밀번호를 입력하는 단순한 로그인으로 느끼게 한다.

## 2. 현재 읽기 상태

### 목적

앱을 열었을 때 "지금 우리가 어디를 읽고 있는지" 바로 알게 한다.

### 요구사항

- 현재 책, 장, section을 보여준다.
- `본문 보기` 버튼은 WOL 신세계역 성경 연구용 해당 장을 새 창으로 연결한다.
- 앱 안에 성서 본문을 저장하거나 노출하지 않는다.
- 현재 segment의 읽음 상태를 보여준다.
- 둘 다 읽었으면 다음 segment로 넘어갈 수 있다.

### 상태 문구

- 아무도 읽지 않음: "함께 읽는 중"
- 나만 읽음: "[상대 이름]을 기다리는 중"
- 상대만 읽음: "[상대 이름]이 먼저 읽었어요"
- 둘 다 읽음: "둘 다 읽었어요"

### 제외

- 강제 날짜 계획표
- 연속 일수
- 전체 퍼센트
- 지각/실패 표현

## 3. 읽음 체크

### 목적

둘 다 읽어야 다음 장으로 넘어가는 구조를 만든다.

### 요구사항

- 사용자는 현재 segment에서 `다 읽었어요`를 누를 수 있다.
- 체크하면 `reading_states.checked_at`이 저장된다.
- 한 사람만 체크한 경우 다음 segment는 아직 활성화하지 않는다.
- 두 사람 모두 체크하면 다음 segment를 활성화한다.
- 이미 체크한 사람에게는 체크 완료 상태를 보여준다.

### 예외

- 과거 segment에도 읽음 상태는 남아 있다.
- 실수로 체크한 경우를 위한 취소는 MVP에서는 제외한다.
- 운영상 필요하면 Supabase Dashboard에서 수정할 수 있다.

## 4. 책 선택과 제안/수락

### 목적

정해진 계획표가 아니라 둘이 책을 골라 읽게 한다.

### 요구사항

- 66권 목록을 section별로 보여준다.
- 현재 읽는 책을 강조한다.
- 책 안에서는 장 순서대로 진행한다.
- 책을 다 읽으면 다음 책을 선택할 수 있다.
- 시작 책은 전도서다.
- 한 사람이 다음 책을 제안할 수 있다.
- 다른 사람이 수락하면 그 책의 1장으로 넘어간다.
- 대기 중인 제안은 오늘 화면과 읽기 화면에 표시한다.

### 제안 상태

- 제안 없음
- 수락 대기 중
- 수락됨
- 취소됨

### 이후 확장

- 제안 이유 메모
- 읽고 싶은 책 목록

## 5. 하이라이트

### 목적

마음에 남는 구절을 segment에 묶어 남긴다.

### 요구사항

- 구절 참조를 입력한다.
- 장 안의 구절 번호를 선택할 수 있다.
- 단일 구절과 구절 범위를 모두 지원한다.
- 색을 선택한다.
- 짧은 메모를 선택적으로 남긴다.
- 작성자 색으로 누가 남겼는지 보여준다.
- 답글과 반응을 달 수 있다.
- 과거 segment에도 하이라이트를 추가할 수 있다.
- 본인 하이라이트는 수정/삭제할 수 있다.

### 색상

기본 선택지:

- 핑크
- 라벤더
- 하늘
- 올리브

작성자 식별은 별도의 진한 색을 쓴다. 하이라이트 배경은 파스텔을 사용한다.

### 제외

- WOL 본문 위에 직접 드래그해서 표시하는 기능
- 구절 원문 저장
- 복잡한 원문 기반 범위 검증

### 구절 번호 선택 UI

- 현재 segment의 장 번호 범위를 기준으로 구절 버튼을 보여준다.
- 구절 하나를 누르면 단일 선택이 된다.
- 시작 구절을 누른 뒤 끝 구절을 누르면 범위 선택이 된다.
- 선택 결과는 `전도서 3:1-8`처럼 `verse_ref`에 저장한다.
- 정확한 절 수 데이터가 없는 책은 MVP에서 직접 입력 fallback을 제공한다.

## 6. 코멘트

### 목적

특정 segment에 묶이는 깊은 생각을 저장한다.

### 요구사항

- segment 상세에서 코멘트를 작성한다.
- 작성자, 작성 시간, 본문을 보여준다.
- 답글을 달 수 있다.
- 작은 반응을 달 수 있다.
- 과거 segment에도 새 코멘트를 추가할 수 있다.
- 본인 코멘트는 수정할 수 있게 한다.

### 정책

- 코멘트에는 읽음 표시를 붙이지 않는다.
- 코멘트에는 접속 중 표시를 붙이지 않는다.
- 채팅처럼 빠른 답장을 요구하는 UI로 만들지 않는다.

### 수정/삭제

- 본인 코멘트는 수정할 수 있다.
- 본인 코멘트는 삭제할 수 있다.
- 삭제는 soft delete로 처리한다.

### MVP 제외 가능

- 긴 서식 편집
- 마크다운
- 이미지 첨부

## 7. 답글과 반응

### 목적

긴 코멘트를 새로 쓰지 않아도 작게 반응할 수 있게 한다.

### 요구사항

- 하이라이트에 답글을 달 수 있다.
- 코멘트에 답글을 달 수 있다.
- 하이라이트, 코멘트, 답글에 반응할 수 있다.
- 같은 사람이 같은 대상에 같은 반응을 중복으로 달 수 없다.
- 본인 답글은 수정/삭제할 수 있다.

### MVP 기본 반응

- 하트 하나로 시작한다.

이후 필요하면 여러 반응을 추가한다.

## 8. 실시간 채팅

### 목적

대체복무 환경에서 둘의 연락 수단 역할을 한다.

### 요구사항

- 메시지를 보낼 수 있다.
- 새 메시지가 실시간으로 나타난다.
- 메시지는 영구 보관한다.
- 날짜 구분선을 보여준다.
- 내 메시지와 상대 메시지를 말풍선으로 구분한다.
- 내 메시지는 현재 사용자 색을 사용한다.
- 상대 메시지는 흰색 또는 연한 배경을 사용한다.

### 반응형

- 모바일: 독립된 전체 화면
- 웹: 버튼으로 여닫는 오른쪽 drawer 또는 floating panel
- 넓은 웹: 읽기 화면 위에서 채팅을 열고 닫을 수 있음

### 제외

- 단체방
- 파일 첨부
- 음성 메시지
- 메시지 검색
- 타이핑 중
- 마지막 접속 시간

### 수정/삭제

- 본인 메시지는 수정할 수 있다.
- 본인 메시지는 삭제할 수 있다.
- 삭제된 메시지는 목록에서 숨기거나 "삭제된 메시지예요" 정도로 조용히 표시한다.
- 삭제는 soft delete로 처리한다.

## 9. 채팅 읽음 표시

### 목적

상대가 메시지를 봤는지 작게 알려준다.

### 요구사항

- 내가 보낸 마지막 메시지 아래에만 작게 표시한다.
- 모든 메시지마다 반복해서 표시하지 않는다.
- 색은 연한 보조 텍스트로 한다.
- 성서 코멘트에는 적용하지 않는다.

### 문구

- "읽음"

강한 알림이나 재촉 문구는 쓰지 않는다.

## 10. 채팅 접속 상태

### 목적

상대가 지금 앱 안에 있는지 부드럽게 느끼게 한다.

### 요구사항

- 채팅 화면에서 상대가 접속 중이면 작은 점 또는 짧은 문구를 보여준다.
- 접속 중이 아니면 크게 표시하지 않는다.
- presence는 Supabase Realtime Presence로 처리한다.

### 문구 후보

- "함께 있는 중"
- 작은 초록/작성자색 점만 표시

### 제외

- 마지막 접속 시간
- 타이핑 중
- 접속/해제 토스트

## 11. 기록

### 목적

시간이 지나 쌓인 기록을 다시 보는 공간을 만든다.

### MVP 요구사항

- 성경 책 목록을 보여준다.
- 책을 누르면 장 목록을 보여준다.
- 각 장 옆에 읽음 상태를 표시한다.
- 장을 누르면 장별 기록 상세로 이동한다.
- 장별 기록 상세에는 읽은 날짜, 코멘트, 하이라이트, 답글, 반응을 보여준다.
- 기록 탭은 열람 전용이다. 새 코멘트 작성, 수정, 삭제는 `코멘트` 탭에서 한다.

### 이후 요구사항

- 사람별 필터
- 날짜별 회고
- 채팅 검색
- 좋아요/반응이 많은 기록

## 12. 알림

MVP에서는 앱 밖 푸시 알림은 넣지 않고, 앱 안에서 확인할 수 있는 조용한 알림만 넣는다.

이유:

- 답장 압박을 줄이는 앱의 원칙과 맞지 않을 수 있다.
- 웹 푸시 설정은 초기 구현 부담이 있다.
- 앱 안 알림만으로도 오늘 쓸 MVP의 변화 확인은 충분하다.

MVP 요구사항:

- 새 채팅 메시지 unread badge
- 새 코멘트/답글 badge
- 다음 책 제안/수락 알림
- 상대가 읽음 체크했을 때 앱 안 알림
- 알림 목록에서 관련 화면으로 이동

MVP 이후 검토:

1. 이메일 알림
2. 웹 푸시

## 13. 관리자/운영

공개 관리자 화면은 MVP에서 만들지 않는다.

초기 운영은 Supabase Dashboard로 한다.

- 사용자 계정 생성
- 현재 책/segment 수정
- 잘못된 데이터 수정
- seed 데이터 확인

앱이 안정되면 주환 전용 작은 운영 화면을 만들 수 있다.

## 비기능 요구사항

### 보존

- 코멘트, 하이라이트, 메시지는 영구 보관한다.
- 삭제는 가능하면 soft delete로 처리한다.

### 보안

- 두 사람만 접근 가능해야 한다.
- Supabase RLS를 켠다.
- service role key는 브라우저에 노출하지 않는다.

### 성능

- 초기 화면은 빠르게 떠야 한다.
- 채팅은 최근 메시지부터 적당히 가져오고, 과거는 필요할 때 더 불러온다.
- 기록이 쌓이면 pagination을 사용한다.

### 접근성

- 버튼은 모바일에서 누르기 충분히 커야 한다.
- 작성자 색만으로 의미를 전달하지 않는다.
- 텍스트 대비는 충분히 확보한다.

## 결정된 것

- 코멘트 수정/삭제는 MVP에 넣는다.
- 채팅 메시지 수정/삭제는 MVP에 넣는다.
- 기록 탭은 첫 배포에 넣는다.
- 구절 번호 선택 UI는 MVP에 넣는다.
- 알림은 앱 안 unread badge와 알림 목록으로 넣는다.
